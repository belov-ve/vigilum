# Vigilum

**Vigilum** — это легковесный сервис на языке Go, предназначенный для непрерывного контроля состояния ресурсов (сервисов) и отправки оповещений о смене их статуса в Telegram/Matrix через [notify-bot](https://github.com/belov-ve/notify-bot).

Сервис поддерживает горячую перезагрузку конфигурации без перезапуска, умеет выполнять базовые сетевые проверки (HTTP, HTTPS, TCP, UDP, Ping) и предоставляет специализированный режим мониторинга панелей управления **Webadmin**.

---

## Основные возможности

* **Разнообразные методы контроля**:
  * **HTTP / HTTPS**: отправка GET-запросов и валидация статус-кодов.
  * **TCP**: проверка открытия портов.
  * **UDP**: отправка пакета и отслеживание ICMP-ответов об ошибке.
  * **ICMP Ping**: проверка доступности узла через системную утилиту `ping`.
  * **Webadmin**: специализированная проверка здоровья управляющей службы и её внутренних подсервисов.
* **Панель управления (Vigilum Admin)**: современный веб-интерфейс для управления файлом конфигурации в реальном времени с поддержкой авторизации по Cookies, резервного копирования и восстановления, поиска, фильтрации и сортировки.
* **Горячая перезагрузка (Hot Reload)**: непрерывное отслеживание изменений файла конфигурации (`config.yaml`) по времени модификации. Изменения вступают в силу «на лету» без перезапуска контейнера.
* **Динамический планировщик**: при изменении конфигурации воркеры для новых или включенных сервисов запускаются автоматически, а для удаленных или выключенных — мгновенно останавливаются с отправкой уведомлений.
* **Эмулятор уведомлений (Mock Bot)**: встроенный эмулятор бота для локального тестирования отправки сообщений без засорения реальных чатов.
* **Встроенный Healthcheck**: HTTP-сервер здоровья контейнера для интеграции с механизмом Docker Healthcheck.
* **Строгое логирование**: использование структурированного пакета `log/slog` с поддержкой 4 уровней логирования (`DEBUG`, `INFO`, `WARN`, `ERROR`).

## Конфигурация приложения

Управление работой Vigilum осуществляется через переменные окружения и конфигурационный файл `config.yaml`.

### 1. Переменные окружения

| Переменная окружения | Описание | Значение по умолчанию |
|-----------------------|----------|-----------------------|
| `CONFIG_PATH`         | Путь к файлу конфигурации YAML | `config.yaml` |
| `POLL_INTERVAL`       | Глобальный интервал между проверками (например, `10s`, `1m`) | `10s` |
| `DEFAULT_RETRIES`     | Дефолтное кол-во повторов при ошибке до отправки DOWN-оповещения | `3` |
| `DEFAULT_RETRY_INTERVAL` | Пауза по умолчанию между повторными попытками проверки | `2s` |
| `NOTIFY_BOT_URL`      | URL-адрес эндпоинта отправки уведомлений (`notify-bot`) | *Обязателен для заполнения* |
| `NOTIFY_BOT_HEALTH_URL`| URL-адрес проверки доступности/здоровья `notify-bot` | Автоматически выводится из `NOTIFY_BOT_URL` (заменой пути на `/health`) |
| `LOG_LEVEL`           | Уровень детализации логов (`DEBUG`, `INFO`, `WARN`, `ERROR`) | `INFO` |
| `HEALTH_PORT`         | Порт встроенного HTTP-сервера для Docker Healthcheck | `8080` |

> [!TIP]
> Все интервалы времени в конфигурации и переменных окружения (например, `POLL_INTERVAL`, `retry_interval`) задаются в стандартном формате длительности Go (например, `10s`, `500ms`, `5m`, `1m30s`). Поддерживаемые единицы измерения: `ms` (миллисекунды), `s` (секунды), `m` (минуты), `h` (часы). Допускается их комбинирование (например, `1m30s`).

### 2. Структура файла `config.yaml` и значения по умолчанию

Файл конфигурации состоит из двух секций: `templates` (шаблоны алармов) и `services` (список контролируемых ресурсов).

  # По умолчанию: "🟢 Сервис {name} восстановил работу"

    name: "Google Public DNS"   # Удобное имя для вывода в оповещениях (Обязательно)

    enabled: true               # Флаг активности мониторинга (Обязательно, bool)
    # retries: 2                # Кол-во повторов (Опционально. По умолчанию: DEFAULT_RETRIES из окружения)
    # retry_interval: "1s"      # Интервал повторов (Опционально. По умолчанию: DEFAULT_RETRY_INTERVAL из окружения)

  - id: "webadmin-panel"
    name: "Webadmin Main"
    type: "webadmin"
    target: "http://webadmin.local:5280"

## Описание базовых проверок с примерами

### HTTP / HTTPS (`type: "http"` / `type: "https"`)
Выполняет HTTP GET-запрос к `target` с ограничением таймаута в 5 секунд.
* **Критерий успешности**: HTTP статус-код ответа строго меньше `400` (любой код `2xx` или `3xx` трактуется как нормальная работа).
* *Пример конфигурации*:
  ```yaml
  id: "my-web-site"
  name: "Main Landing Page"
  type: "https"
  target: "https://example.com/healthz"
  enabled: true
  ```

### TCP (`type: "tcp"`)
Пытается установить низкоуровневое TCP-соединение с узлом по схеме `хост:порт`.
* **Критерий успешности**: Успешное выполнение фазы трехстороннего рукопожатия (TCP handshake) в течение 5 секунд.
* *Пример конфигурации*:
  ```yaml
  id: "postgres-db"
  name: "Database Port"
  type: "tcp"
  target: "db.local:5432"
  enabled: true
  ```

### UDP (`type: "udp"`)

Протокол UDP является протоколом без установления соединения (connectionless), что делает классическое сканирование портов ненадёжным. Для решения этой проблемы Vigilum реализует два режима работы: встроенные «умные» протокольные проверки и проверки кастомных портов пустыми пакетами.

#### 1. Правила заполнения UDP-проверок (Активация встроенных протоколов)

Чтобы Vigilum мог достоверно проверить UDP-сервис, конфигурация должна отвечать следующим правилам:
* **Формат Target**: Поле `target` должно обязательно содержать адрес и порт в формате `хост:порт` (например, `192.168.1.100:53` или `dns.google:53`).
* **Активация «умных» протоколов**: Встроенные протоколы прикладного уровня активируются автоматически **на основе порта**, указанного в `target`. Поддерживаются следующие порты:
  * **Порт `53` — DNS**: Отправляет корректный DNS-запрос для домена `google.com` (тип A, класс IN) и ожидает любой валидный ответ от DNS-сервера.
  * **Порт `123` — NTP**: Отправляет 48-байтный клиентский запрос времени (NTP client request, режим `3`) и ожидает ответ от сервера синхронизации времени.
  * **Порт `3478` — STUN**: Отправляет 20-байтный запрос `Binding Request` (используется для проверки работоспособности WebRTC/VoIP серверов, например Coturn) и ожидает ответа.
  * **Порт `161` — SNMP**: Отправляет SNMP v1 `GetRequest` для OID `1.3.6.1.2.1.1.1.0` (системное описание `sysDescr`) с community `public` и ожидает ответ от SNMP-агента.
  * **Порт `5060` — SIP**: Отправляет текстовый запрос `OPTIONS` (пинг телефонии) и ожидает ответ от SIP-сервера (например, `200 OK`, `401 Unauthorized` или другой SIP-статус).

> [!IMPORTANT]
> Если вы хотите использовать встроенную проверку прикладного уровня, вы **обязаны** использовать для службы её стандартный порт. Если служба работает на нестандартном порту (например, DNS на `5353`), Vigilum выполнит обычную проверку кастомного порта.

#### 2. Логирование кастомных портов и принцип их работы

Если порт в `target` отличен от вышеперечисленных, Vigilum переключается в режим **кастомного UDP-порта**:
* **Метод проверки**: На указанный адрес отправляется пустой датаграммный пакет (`[]byte("")`). Цель отправки — спровоцировать целевую ОС вернуть ICMP-сообщение об ошибке (например, `Destination Port Unreachable`), если порт закрыт.
* **Логирование при запуске (WARN)**: Поскольку надежность такой проверки зависит от сетевого окружения, при инициализации или горячей перезагрузке воркера для кастомного UDP-порта Vigilum выводит предупреждающее сообщение (`WARN`) в системный лог:
  * *Пример лога с указанием порта*: 
    `time=... level=WARN msg="Сервис использует кастомный UDP-порт. Проверка выполняется пустым пакетом; ее достоверность зависит от доступности ICMP-ответов на целевом хосте." service=my-custom-service port=12345`
  * *Пример лога без указания порта* (если порт не распарсился): 
    `time=... level=WARN msg="Сервис использует UDP-проверку без явного указания стандартного порта. Проверка выполняется пустым пакетом." service=my-custom-service target=my-host`
* **Ограничения метода**:
  1. Если целевой хост защищен брандмауэром (firewall), который молча блокирует (drop) входящие пакеты и не отправляет ICMP-ответы, то закрытый порт будет ложно определяться как открытый (`Online`), так как таймаут чтения не произойдет, если не отправлять и не читать пакеты (хотя в Vigilum таймаут чтения на пустом пакете без ICMP-ответа теперь строго трактуется как ошибка).
  2. Если на целевом хосте порт закрыт, но ОС возвращает ICMP `Port Unreachable`, Vigilum получит системную ошибку (например, `read udp ...: connection refused`) и переведет сервис в `🔴 Offline`.

#### 3. Таймаут и отсутствие Ping-fallback

* **Таймаут**: На ожидание ответа (или ICMP-сообщения об ошибке) выделяется ровно **1.5 секунды** (`1500ms`). Этого времени достаточно, чтобы ICMP-сообщение о недоступности порта вернулось обратно даже через медленные сети.
* **Отключение Ping-fallback**: В предыдущих версиях при таймауте чтения UDP выполнялся пинг (ICMP ping) хоста, и если хост пинговался, порт считался открытым. В версии **1.1.0** этот механизм **полностью отключен**, поскольку он приводил к ложноположительным статусам `Online` для любых закрытых UDP-портов на живых хостах. Теперь при таймауте или ошибке чтения сервис гарантированно переходит в `🔴 Offline`.

* *Пример конфигурации для встроенного протокола (NTP)*:
  ```yaml
  id: "ntp-server"
  name: "NTP Time Sync"
  type: "udp"
  target: "pool.ntp.org:123"
  enabled: true
  ```

* *Пример конфигурации для кастомного UDP*:
  ```yaml
  id: "custom-udp-service"
  name: "Custom Game Server Port"
  type: "udp"
  target: "game.local:27015"
  enabled: true
  ```

### Ping (`type: "ping"`)
Осуществляет системный запуск утилиты `ping` для отправки одного ICMP Echo Request пакета к целевому хосту.
* **Критерий успешности**: Успешное получение эхо-ответа (код завершения команды `ping` равен `0`).
* *Пример конфигурации*:
  ```yaml
  id: "main-router"
  name: "Gateway Ping"
  type: "ping"
  target: "192.168.1.1"
  enabled: true
  ```

### Webadmin (`type: "webadmin"`)
Специализированный режим мониторинга панелей управления (например, X-UI или аналогичных), содержащих группу внутренних подсервисов. Vigilum контролирует как доступность самой панели, так и состояние каждого подсервиса по отдельности.

#### Алгоритм проверки:
1. **Проверка общей доступности панели**:
   Выполняется HTTP GET-запрос к эндпоинту `{target}/api/system/status` (без авторизации).
   * **Критерий успешности**: Статус-код `200 OK` и JSON-ответ с полями `status.active_state` (ожидается `"active"`) и `status.sub_state` (ожидается `"running"`). При сбое фиксируется недоступность всей панели.
2. **Авторизация и получение сессии**:
   Для доступа к детальным конфигурациям подсервисов выполняется POST-запрос на `{target}/api/auth/login` с логином (`username`) и паролем (`password`) из файла конфигурации.
   * **Кэширование сессии**: Полученные Cookies сохраняются в сессионном `cookiejar` клиента. При последующих опросах сессия используется повторно.
   - Если при обращении к API возвращается код `401 Unauthorized` или редирект (`302`/`303`/`307`), выполняется повторный вход в систему и запрос повторяется.
3. **Отслеживание изменений внутренних подсервисов**:
   Выполняется запрос к списку конфигураций по адресу `GET {target}/api/configs`.
   Для каждого обнаруженного в ответе элемента (`item`) Vigilum контролирует следующие параметры:
   - **Состояние работоспособности (Health)**: поля `status.active_state` (ожидается `"active"`) и `status.sub_state` (ожидается `"running"`). Любое изменение состояния здоровья логируется и отправляет оповещения `🔴` (шаблон `down`) или `🟢` (шаблон `up`). Имя подсервиса в уведомлениях формируется в формате: `{Имя Webadmin} / {Имя подсервиса} ({Имя systemd-службы})`.
   - **Флаг автозапуска (Enabled)**: поле `item.enabled` (статус systemd службы в системе). При его изменении отправляются уведомления `🔵` (шаблон `webadmin_enabled` - Включен автозапуск) или `🟡` (шаблон `webadmin_disabled` - Отключен автозапуск). Обратите внимание: изменение автозапуска не останавливает непрерывный контроль работоспособности подсервиса.
   - **Добавление/Удаление сервисов**: при появлении новой службы отправляется оповещение о начале мониторинга. При удалении службы из панели отправляется оповещение `🟡` (шаблон `disabled`).

## Панель управления (Vigilum Admin)

**Vigilum Admin** — это современная веб-панель управления, позволяющая администрировать список контролируемых ресурсов и шаблоны уведомлений в реальном времени без необходимости вручную редактировать YAML-файл на сервере. Панель работает как SPA (Single Page Application) и общается с Go-бэкендом через REST API.

### 1. Основной функционал интерфейса
* **Два режима отображения списка**:
  * **Grid (Сетка)** — наглядные карточки с полной информацией о сервисе, его типе, адресе и параметрах повторных проверок.
  * **Table (Таблица)** — компактное табличное представление, идеально подходящее для одновременного просмотра большого количества хостов.
  * *Выбранный вид сохраняется в локальном хранилище браузера (`localStorage`) и автоматически применяется при повторном открытии панели.*
* **Интеллектуальная сортировка и поиск**:
  * Сортировка по умолчанию (в порядке расположения в YAML-файле), по имени (A-Z / Z-A), по ID, типу или активности (сначала включенные). Выбор сортировки кэшируется в `localStorage`.
  * Быстрый интерактивный поиск по имени, идентификатору, типу проверки или целевому адресу с выводом счетчика найденных элементов.
* **Двухрежимное сохранение настроек**:
  * **Мгновенное переключение** — тумблер включения/выключения мониторинга в карточке или строке сервиса отправляет запрос `POST /api/services/toggle` и сохраняет изменения в файл немедленно.
  * **Отложенное сохранение** — операции добавления нового сервиса, редактирования параметров, удаления, а также редактирование шаблонов уведомлений сначала фиксируются локально в браузере. Кнопка **Save Configuration** начинает пульсировать янтарным цветом, предупреждая о несохраненных изменениях. При ее нажатии конфигурация записывается на сервер.
* **Резервное копирование и восстановление (Backup & Restore)**:
  * Кнопка **Backup** позволяет в один клик скачать текущий рабочий конфигурационный файл `config.yaml` на локальный компьютер.
  * Кнопка **Restore** позволяет загрузить сохраненный YAML-файл, автоматически валидировать его структуру на бэкенде и применить новые параметры.

### 2. Безопасность и управление сессиями
Для защиты панели от несанкционированного доступа реализован безопасный сессионный механизм:
* **Авторизация по Cookies**: вместо устаревшей Basic Auth, передающей логин и пароль в заголовках каждого запроса в base64, используется сессионная кука `session_token`.
* **Флаги безопасности сессии**:
  * `HttpOnly` — защищает сессионный токен от кражи через XSS-атаки и чтение из JS-скриптов.
  * `SameSite=Strict` — предотвращает межсайтовую подделку запросов (CSRF), запрещая передачу куки при внешних переходах.
  * `Path=/` — кука ограничена только эндпоинтами панели администрирования.
* **Срок действия и автопродление**: время жизни сессии ограничено 12 часами с механизмом «скользящего окна» (sliding window) — при каждом запросе активного пользователя время жизни сессии автоматически продлевается.
* **Защита бэкенда**:
  * Активные сессии хранятся в оперативной памяти сервера (in-memory map) с потокобезопасным доступом через `sync.RWMutex`.
  * Валидация учетных данных выполняется с использованием `crypto/subtle.ConstantTimeCompare` для предотвращения Timing-атак (атак по времени).
  * При логауте токен удаляется из памяти сервера, а кука очищается в браузере. В случае получения ошибки `401 Unauthorized` интерфейс мгновенно перенаправляет пользователя на форму авторизации.

## Развертывание через Docker Compose (совместный запуск)

Для полноценного использования системы рекомендуется разворачивать связку из демона мониторинга `vigilum` и панели управления `vigilum-admin`. Они используют общий файл конфигурации `config.yaml`, изменения в котором мгновенно подхватываются демоном `vigilum` на лету благодаря механизму Hot Reload.

### Готовый файл `docker-compose.yml`

Создайте в рабочей директории файл `docker-compose.yml` со следующим содержимым:

```yaml
version: '3.8'

services:
  # Основной демон мониторинга Vigilum
  vigilum:
    image: vigilum:1.3.0
    container_name: vigilum
    restart: unless-stopped
    volumes:
      - ./config.yaml:/app/config.yaml
    environment:
      - CONFIG_PATH=/app/config.yaml
      - POLL_INTERVAL=10s                             # Интервал проверок ресурсов
      - LOG_LEVEL=INFO                                # Уровень логирования (DEBUG, INFO, WARN, ERROR)
      - DEFAULT_RETRIES=3                             # Количество попыток до отправки аварии
      - DEFAULT_RETRY_INTERVAL=2s                     # Пауза между повторными попытками
      - NOTIFY_BOT_URL=http://notify-bot.local/notify # URL вашего реального notify-bot
      - TZ=Europe/Moscow                              # Настройка часового пояса для вывода даты/времени
    ports:
      - "18080:8080"                                  # Порт для Docker Healthcheck

  # Панель управления конфигурациями Vigilum Admin
  vigilum-admin:
    image: vigilum-admin:1.3.0
    container_name: vigilum-admin
    restart: unless-stopped
    volumes:
      - ./config.yaml:/app/config.yaml               # Общий монтируемый файл конфигурации
    environment:
      - CONFIG_PATH=/app/config.yaml
      - PORT=8080                                     # Внутренний порт приложения
      - ADMIN_USERNAME=admin                          # Логин для входа в веб-панель
      - ADMIN_PASSWORD=admin                          # Пароль для входа в веб-панель (измените на надежный!)
      - VIGILUM_API_URL=http://vigilum:8080           # URL-адрес демона Vigilum внутри Docker-сети
      - TZ=Europe/Moscow                              # Настройка часового пояса событий
    ports:
      - "18082:8080"                                  # Внешний порт панели администрирования
```

### Запуск и управление

1. **Запуск сервисов**:
   ```bash
   docker compose up -d
   ```
2. **Просмотр логов демона мониторинга**:
   ```bash
   docker compose logs -f vigilum
   ```
3. **Просмотр логов панели управления**:
   ```bash
   docker compose logs -f vigilum-admin
   ```
4. **Проверка работоспособности**:
   * Панель администрирования доступна в браузере по адресу: `http://localhost:18082` (введите логин и пароль, указанные в `docker-compose.yml`).
   * Проверить встроенный healthcheck демона Vigilum можно командой:
     ```bash
     curl http://localhost:18080/health
     ```

### Контроль приложения внешними системами мониторинга

Опубликованный порт `18080` демона `vigilum` позволяет легко подключить контроль работоспособности самой системы мониторинга к внешним сервисам (например, **Uptime Kuma**, **Zabbix** или **Prometheus**):
* **Uptime Kuma**: настройте HTTP-монитор на адрес `http://<IP_хоста>:18080/health` с ожиданием статус-кода `200`. Это гарантирует, что при падении или зависании основного контейнера `vigilum` вы незамедделительно получите оповещение.
* **Пример ответа Healthcheck**:
  ```json
  {"status":"healthy","time":"2026-05-29T22:39:00+03:00"}
  ```

## Тестирование с помощью эмулятора mock-bot

Для локального тестирования Vigilum без отправки спама в реальные каналы связи в проекте предусмотрен эмулятор — **mock-bot**.

### 1. Запуск эмулятора
Эмулятор представляет собой мини-сервер на Go. Запустите его локально:
```bash
cd build && PORT=18043 go run ./mock-bot/main.go
```
*Лог запуска*:
`time=... level=INFO msg="Запуск mock-bot сервера для локального тестирования" port=18043`

### 2. Запуск Vigilum с перенаправлением на эмулятор
Задайте адрес эмулятора в переменной окружения `NOTIFY_BOT_URL` и запустите Vigilum:
```bash
cd build && NOTIFY_BOT_URL=http://localhost:18043/notify CONFIG_PATH=../config.yaml go run ./app/
```

### 3. Симуляция событий
* **Включение мониторинга**: при старте Vigilum или при изменении конфига в консоли эмулятора мгновенно отобразятся системные сообщения:
  ```text
  === [Mock Bot] ПОЛУЧЕНО НОВОЕ УВЕДОМЛЕНИЕ ===
  Текст:  🔵 Включен мониторинг сервиса Google Public DNS
  =============================================
  ```
* **Имитация аварии**: временно отключите проверяемый TCP-сервис или заблокируйте доступ. После прохождения заданного количества повторов (`retries`) в консоли появится:
  ```text
  === [Mock Bot] ПОЛУЧЕНО НОВОЕ УВЕДОМЛЕНИЕ ===
  Текст:  🔴 Сервис Local Port недоступен! Ошибка: dial tcp 127.0.0.1:8081: connect: connection refused
  =============================================
  ```
* **Имитация восстановления**: запустите проверяемый сервис заново. На следующем цикле опроса эмулятор выведет:
  ```text
  === [Mock Bot] ПОЛУЧЕНО НОВОЕ УВЕДОМЛЕНИЕ ===
  Текст:  🟢 Сервис Local Port восстановил работу
  =============================================
  ```

## Развертывание через Docker Compose (для запуска)

Для запуска Vigilum в фоновом режиме на сервере используется готовый собранный Docker-образ. Сборка не требуется, образ скачивается из реестра (или используется локально собранный `vigilum:latest`).

Пример готового файла конфигурации `docker-compose.yml` для запуска:

```yaml
version: '3.8'

services:
  vigilum:
    image: vigilum:latest        # Ссылка на собранный образ Vigilum
    container_name: vigilum
    restart: unless-stopped
    # Монтируем файл конфигурации из текущей папки хоста внутрь контейнера
    volumes:
      - ./config.yaml:/app/config.yaml
    environment:
      - CONFIG_PATH=/app/config.yaml
      - POLL_INTERVAL=15s                             # Интервал проверок
      - DEFAULT_RETRIES=3                             # Кол-во попыток до аларма
      - DEFAULT_RETRY_INTERVAL=2s                     # Пауза между попытками
      - LOG_LEVEL=INFO                                # Уровень логирования
      - NOTIFY_BOT_URL=http://192.168.1.50:8043/notif # URL вашего реального notify-bot
    ports:
      - "18080:8080"                                  # Порт для хелсчеков (публикация для внешних систем)
```

Для запуска выполните стандартную команду:
```bash
docker compose up -d
```
Для мониторинга логов:
```bash
docker logs -f vigilum
```
Для ручной проверки работоспособности контейнера извне (healthcheck):
```bash
curl http://localhost:18080/health
```

### Контроль приложения внешними системами мониторинга

Опубликованный порт `18080` (публикация не обязательна, а при необходимости) позволяет подключить контроль работоспособности самого Vigilum к внешним системам мониторинга (например, **Uptime Kuma**, **Zabbix**, **Nagios** или **Prometheus**):

1. **Uptime Kuma**:
   - Тип монитора: `HTTP(s)`.
   - URL: `http://<IP_хоста_докера>:18080/health`
   - Ожидаемый статус: `200`
   - Это гарантирует, что если контейнер `vigilum` завершит работу с ошибкой или зависнет, внешняя система мониторинга пришлет вам уведомление о его недоступности.
2. **Prometheus / Nginx Reverse Proxy**:
   - Вы можете проксировать эндпоинт `/health` наружу под защитой basic-auth или использовать его напрямую внутри закрытой сети администрирования.
   - Пример возвращаемого ответа:
     ```json
     {"status":"healthy","time":"2026-05-29T10:00:00+03:00"}
     ```
